## Overview

This project builds a domain-specific conversational AI system designed to answer
questions about BCA individual banking products. The system is built on the
Retrieval-Augmented Generation (RAG) architecture, which combines a vector search
engine with a large language model (LLM) to produce answers that are grounded in
actual product documentation rather than relying solely on the model's pre-trained
knowledge.

The chatbot covers over 80 BCA individual products across five categories: savings
accounts, individual loans, wealth management (insurance and investment), credit
cards, electronic money, and rewards. Users can ask questions in natural language
about product requirements, fees, interest rates, features, and benefits, and the
system retrieves the most relevant product chunks before generating a structured
response.

## System Architecture

The system is built on four main components working in sequence:

**1. Knowledge Base — JSON Product Documents**

Product information for all BCA individual products was structured into a JSON
file (`bca_produk_individu.json`) containing text chunks with associated metadata
fields including product name, category, and source section. Each chunk represents
a discrete unit of product information such as a feature description, a list of
requirements, or a fee schedule.

**2. Embedding and Vector Indexing — BGE-M3 + Pinecone**

Each text chunk was encoded into a dense vector representation using the
`BAAI/bge-m3` multilingual embedding model from SentenceTransformers. BGE-M3 was
selected because it supports both Indonesian and English text, which is relevant
given that BCA product documentation is primarily in Indonesian.

Vectors were upserted into a Pinecone serverless index in batches of 100, with
metadata preserved alongside each vector. The metadata includes the product name
and the original text, which allows the retrieval layer to filter results by
product when the user's query refers to a specific product.

**3. Retrieval Layer — Pinecone Similarity Search + Product Filter**

When a user submits a query, the system first runs an intent analysis step using
LLaMA 3.1 to extract the product name being referenced. The query is then embedded
using the same BGE-M3 model with the `query:` prefix (following the asymmetric
retrieval convention for BGE models) and used to retrieve the top 20 most similar
chunks from Pinecone.

If a specific product name was identified in the intent analysis step, the
retrieved chunks are filtered to retain only those matching that product. If no
matching product chunks are found after filtering, the system falls back to the
full unfiltered result set. The retrieved context is then truncated to a maximum
of 6,000 characters before being passed to the language model.

**4. Generation Layer — LLaMA 3.1 via Groq API**

The final answer is generated by LLaMA 3.1 (8B Instant) served through the Groq
API. The model receives a structured prompt containing the retrieved product
context and the user's question. The system prompt instructs the model to answer
only from the provided context and to avoid generating information that is not
present in the retrieved documents.

## Intent Classification and Query Routing

Before retrieval, each user query passes through two classification functions:

**List Query Detection (`is_list_query`)**

This function checks whether the user is asking for a list of products within a
specific category (e.g., "what are all the savings products at BCA?") rather than
asking about a specific product's details. If the query matches a list intent and
does not contain detail-specific keywords (such as "requirements", "interest rate",
"fee", or "benefit"), the system bypasses the retrieval pipeline entirely and
returns a pre-formatted product list for the relevant category directly.

This prevents unnecessary vector search calls and ensures consistent, complete
responses for category-level listing queries.

**Intent Analysis (`analyze_intent`)**

For all other queries, the intent analysis function calls LLaMA 3.1 with a
structured prompt to extract the product name being referenced and to determine
whether the query requires clarification. The output is parsed as a JSON object
containing:

- `product_name` — the specific BCA product identified in the query
- `needs_clarification` — a boolean indicating whether the query is ambiguous
- `clarification_question` — a follow-up question to resolve the ambiguity
- `clarification_options` — a list of candidate products to present to the user

If the model identifies that the user's query is ambiguous (for example, asking
about "kredit" without specifying whether they mean a vehicle loan, home loan, or
personal loan), the system returns the clarification question and presents options
before proceeding to retrieval.

## Limitations

The current system operates on a static JSON knowledge base. Product information
that changes after the knowledge base was compiled — such as updated interest rates,
new product launches, or revised requirements — will not be reflected in the
chatbot's responses without re-running the data preparation and indexing pipeline.

The system also does not maintain conversation history across turns in the current
implementation. Each query is treated independently, which means the chatbot cannot
handle follow-up questions that depend on the context of previous turns.
